THE INTEGRATION

Sovereign Mode wraps the calls your product already makes. Your prompts, retrieval, and agents run unchanged; the wrapper decides where they run and who can read what goes in.

import { sovereign } from "@ember/sdk";

// Wrap the calls your product already makes. Nothing else changes.
const session = await sovereign.open({ customer });

// Runs content-pinned in the sealed environment. Plaintext never reaches you.
const output = await session.run(pipeline, input);

// Nothing persists. The receipt names what ran, where, under whose key.
const receipt = await session.close();

The customer’s key is a passkey: one tap to switch the toggle on, one recovery phrase saved. No wallet, nothing to trade.

THE VERIFICATION

Before the first corporate payload enters the environment, the buyer’s security team verifies it independently. Three checks, none of which require trusting us or you:

  • Pull the attestation quote. Every sovereign session exposes one.
  • Verify the quote’s signature against the silicon vendor’s published root of trust. This proves genuine attested hardware, not an emulator.
  • Compare the measurement in the quote to the content-pinned hash of the audited build, and to the session’s receipt on the ledger. Equal hashes prove the audited code is exactly what ran.
# the attestation quote for any sovereign session
curl https://api.embersovereignty.com/attestation/{session_id}

The receipt trail repeats these facts for every session, so drift is visible the moment it happens. What a reviewer audits once, the architecture re-proves continuously.
